# 2026-05-15

## Issue #41 — Weekly view does not show newly-created note

### Problem

User reports: clicking **+ Create** on a weekly "missing" card creates the file on disk but the view keeps showing the missing card. Clicking again surfaces a "Note already exists" notice. Daily view works fine.

### Root cause

`detectPeriodType()` in `src/utils/periodic-note-utils.ts` classified files by iterating period types in fixed order (`daily, weekly, …`) and using `file.path.startsWith(config.folder)` with no path-segment boundary check.

When the daily folder is a parent of the weekly folder (e.g. daily=`Journal`, weekly=`Journal/Weekly`), the daily branch matches first and every weekly file is returned as `'daily'`. The weekly view then filters those files out, so the newly-created note never appears.

A secondary defect: `Journal` would also match `JournalArchive/...` because `startsWith` doesn't enforce a path separator.

### Fix

`detectPeriodType()` now:

1. Collects all enabled period configs with non-empty folders.
2. Sorts by folder length descending so the most specific folder wins.
3. Uses a new `isPathInFolder()` helper that requires the path to equal the folder or to start with `folder + '/'`.

`done-reviews-utils.ts` already normalized with a trailing slash and was not affected.

### Tests

New `src/utils/periodic-note-utils.spec.ts` (10 tests, non-regression for issue #41):

- Disabled and folder-less configs are skipped.
- Most-specific folder wins regardless of period-type ordering (five-level nesting).
- `JournalArchive/...` is not classified as belonging to `Journal`.
- Files directly under a nested folder are still detected.
