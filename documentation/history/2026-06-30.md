# 2026-06-30

## Fix: custom date formats not detected as existing notes (#42)

**Problem**: With custom periodic-note formats — Daily `YYYY-MM-DD-dddd`
(weekday name) and Monthly `YYYY-MM-MMMM` (month number + month name) — existing
notes were never recognized in Periodic Review: every entry rendered grey/missing.
Standard formats (`YYYY-MM-DD`) at the same folder depth worked, so the folder
nesting was a red herring. The "Create" button on a greyed entry, however, opened
the already-existing note ("Note already exists"), so the plugin _could_ find the
file one way but not the other.

**Root cause** (single, shared by both reported bugs): `parseDateFromFormat`
(`src/utils/date-utils.ts`) returned `null` for these formats. date-fns `parse`
rejects two token combinations that moment.js accepts:

1. A weekday-name token converted to the **local** `eeee`/stand-alone `cccc`
   family throws `mustn't contain 'yyyy' and 'eeee' at the same time` when combined
   with the calendar-year token `yyyy` (local-week-dependent day tokens require a
   week-year context).
2. Duplicate tokens for the same unit — month-number `MM` + month-name `MMMM` —
   throw `mustn't contain 'MM' and 'MMMM' at the same time`.

When parsing returned `null`, existence detection (`extractDateFromNote` →
`getAvailableDatesForColumn`'s `dateEntryMap`) never associated files with dates →
grey. The "Create" path uses **forward** `formatDate` (no such restriction), which
is why it found the file — Bug 2's inconsistency is the same root cause seen from
the other direction.

**Fix** (`src/utils/date-utils.ts`):

- `convertMomentToDateFns`: weekday names now convert to the **formatting** `E`
  family (`dddd`→`EEEE`, `ddd`→`EEE`, `dd`→`EEEEEE`) instead of `eeee`/`eee`/
  `eeeeee`. `EEEE` is week-year-independent, so it parses fine alongside `yyyy` and
  produces byte-identical output when formatting. This fixes the daily case on the
  direct date-fns path (no fallback needed).
- `parseDateFromFormat`: added a tolerant regex fallback
  (`parseDateFromFormatTolerant`) that runs **only when the direct date-fns parse
  throws or returns invalid**, so the well-tested happy path is untouched and the
  change can only improve on the previous `null`. It builds a regex from the moment
  format, captures the determinant numeric components (year / week-year / month-num
  or month-name / day / week / quarter / day-of-year), ignores decorative tokens
  (weekday names; a month _name_ is discarded when a numeric month is present), and
  reconstructs the date — reusing date-fns only for ISO-week math. Out-of-range
  components are round-trip-validated so a malformed filename (`2026-13-99`) is not
  silently rolled over into a phantom date.

**Tests**: 9 new cases under `parseDateFromFormat` → "issue #42 custom formats"
(weekday/short-weekday daily, month-name/short-month monthly, reordered
month-name-only, weekday-doesn't-shift-day, format↔parse round-trip, out-of-range
rejection, non-matching filename). Updated the 3 conversion tests that asserted the
old `eeee` output. Suite: 266 pass (was 257).

**Verification**: `bun run tsc` (0), `bun run lint` (0 warnings), `bun test`
(266 pass), `bun run build` all green. Runtime check still required in a live vault:
set Daily format `YYYY-MM-DD-dddd` / Monthly `YYYY-MM-MMMM`, confirm existing notes
render white (detected) in Periodic Review and that re-navigating keeps them white.

**Decisions / open items**:

- The fallback is intentionally general (handles arbitrary token order and the
  realistic conflict classes) but scoped to date-determinant tokens; time tokens in
  periodic-note filenames are out of scope.
- The issue's **feature request** ("option to collapse frontmatter by default when
  opening a note from Periodic Review") is a separate enhancement and is **not**
  addressed here.
