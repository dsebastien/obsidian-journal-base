# 2026-06-20

## Fix: review-view note scroll snap-back (#issue: scroll bug)

**Problem**: In the Periodic Review view, scrolling inside a column's note editor
scrolled a little then snapped back to the top, and a second scrollbar misbehaved.

**Root cause**: Two nested vertical scroll containers. `.pr-column__content`
(`flex-1 overflow-y-auto`) was the real scroller; the embedded CodeMirror
`.cm-scroller` (`overflow-auto`) had no bounded height because `.cm-editor` grew to
full content height, so the inner scroller was inert. CodeMirror's measure/layout
pass calls `scrollIntoView` for the cursor, which walks up to the first
`overflow:auto` ancestor (`.pr-column__content`) and yanks it back to the top —
worsened by periodic `refreshContent()` dispatches on data updates.

**Fix** (CSS only, `src/styles.src.css`): scoped to `.periodic-review-view`, build a
flex-column chain from `.pr-column__content` → `.pn-card` → `.pn-card__content` →
`.pn-card__editor` → `.cm-editor` so the CodeMirror `.cm-scroller` becomes the SOLE
vertical scroll region (`min-h-0` on each flex child; header is `flex-none`; padding
moved onto `.cm-scroller`). The outer `.pr-column__content` is now `overflow-hidden`,
so CM's `scrollIntoView` can never reach it. Mobile `@media (max-width: 599px)`
reverts the chain because the stacked mobile layout relies on the outer container
scrolling within its `max-height: 50vh` cap. No JS change needed. The stacked
Periodic Notes view is untouched (different container class).

**Verification**: tsc, lint (0 warnings), 222 tests, and prod build all pass.
Runtime (live Obsidian) check still required: scroll a long note in the review view
and confirm (1) no snap-back, (2) a single scrollbar in the editor, (3) mobile still
scrolls within the 50vh cap.

**Method**: diagnosed/verified via a 3-angle workflow (CSS layout, CodeMirror scroll
model, JS refresh-jitter) plus an adversarial verify pass.
